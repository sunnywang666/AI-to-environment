# 自查与迭代方案 v2（止损 · 理顺）

> 目的：停止反应式打补丁，先把问题盘清，再用**有序的几步**一次性理顺。
> 前提坦白：我这边**没有浏览器、看不到渲染**，一直凭代码盲改——这是反复返工、让你疲惫的根本原因。下面的方案专门为此设计：把你的负担降到"只在 2-3 个检查点看一眼截图"。

## 一、元问题（为什么这么累）

1. **没有先锁设计系统**就逐段开工 → 每段各做各的，间距/字号/配色/图表样式都不统一 → 整体"散、土"。
2. **反应式打补丁** → 改 A 碰坏 B，补丁层层叠加，没人通盘看。
3. **我看不到效果** → 很多视觉 bug 要靠你一次次截图指出，本该我自己发现。

> 结论：不是某一个功能坏了，是**缺一套统一规范 + 缺验证手段**。先补这两样，后面就顺了。

## 二、自查清单（我能从代码里确认的问题）

### A. 一致性 / 排版（最影响"散"的感觉）
- 各段**节奏失衡**：有的段就几行字，有的段是 620vh / 340vh 的超长 pin 场景，滚动体验忽长忽短。
- **图表样式不统一**：排行图、杰文斯折线、等量图、地图——标题/来源/图例/配色各写各的，没有统一"图表组件规范"。
- **字号无统一阶梯**：大数字、标题、正文、注释多处 clamp 各取值，不成体系。
- **间距不统一**：段落 padding、图与文间距各段不同。

### B. 第 2 段 3D（最折腾的一块）
- 粒子"从少到多"的数量是**我拍的**（450→2900→4200），并非真实挂钩，概念上仍站不住——你早就指出过。
- 620vh 一段太长，"爬升→停顿"机制新但容易让人**不知道发生了什么 / 卡顿感**。
- 4 个模型是程序化糙建模，精度有限。
- **它到底要不要这么重**？值得重新判断——可能"克制一点"反而更好。

### C. 第 3 段地图（风险最高、我没法验证）
- **全屏 sticky + 负 margin** 这套布局我无法验证，很可能在你浏览器**错位/抖动**。
- 中美双图并排在全屏下各占一半、偏小；"飞向缩放"可能别扭。
- 文字直接浮在地图上的**可读性**没把握。

### D. 过渡
- 现在有三套过渡机制（hero→暗带、墨↔纸带、逐块渐隐）——**能用但不成体系**，观感不连贯。

### E. 文案 / 数据
- v2 文案不错，但**页面文字与文档可能已漂移**；部分数字是示意/预测，缺统一的"实测/预测/来源"标注规范。
- 新的 192 国数据集、电价数据还没织进去。

### F. 性能 / 移动端 / 可访问性
- Three.js + 多个 canvas + D3 地图 + GSAP 同时加载，多个 rAF 常驻，**弱机可能卡**。
- **移动端基本没做**（全屏、sticky、3D、双图很可能在手机上垮）。
- 可访问性几乎为零。

## 三、迭代方案（分阶段，每阶段一个检查点）

> 原则：**先统一、再打磨、后扩展**。每个阶段结束我给你一组"该看哪几屏"的清单，你只需截图反馈一次。

### 阶段 0 · 锁设计系统（不写功能，半天）
产出一份 `设计系统.md` + 一套 tokens：
- 字号阶梯（4-5 级）、间距阶梯（一套 spacing scale）
- 配色使用规则（什么时候用琥珀/青/红，克制 glow）
- **统一图表组件规范**（标题/来源/图例/坐标轴一个样式）
- 统一运动规则（一种缓动、一种进出场）
→ **检查点 0**：你看规范文档，拍板。

### 阶段 1 · 一致性大扫除（把现有东西套进规范）
- 全段统一字号/间距/图表样式/来源标注。
- 收敛过渡：保留一种主过渡，去掉杂乱叠加。
- **决断第 3 段地图**：要么把全屏 sticky 调稳、要么降级为更简单可靠的版式（我倾向先稳）。
→ **检查点 1**：你通览一遍截图，标红还散的地方。

### 阶段 2 · 只打磨 2 个"招牌交互"，其余做干净
- 选定 2 个招牌（建议：**开场聊天框** + **第 5 段口径滑块**，最稳最点题）。
- 第 2 段 3D 和第 3 段地图：**要么打磨到稳、要么果断简化**，不留半成品。
→ **检查点 2**：确认招牌交互手感。

### 阶段 3 · 内容与数据定稿
- 文案与页面对齐；用 192 国数据集 + 电价数据校准每个数字；统一实测/预测/来源标注。

### 阶段 4 · 响应式 + 性能 + 可访问性收尾
- 移动端降级、按需启停 rAF、图片懒加载、alt/语义。
